# Design_Rationale.md: เอกสารอธิบายเหตุผลในการเลือกใช้ UX/UI
**Project:** Real-time Multi-user To-Do List Application

---

## 1. ปรัชญาการออกแบบ (Design Philosophy)
การออกแบบยึดหลัก **"Minimalism & Efficiency"** เนื่องจากเป็นเครื่องมือที่เน้นการใช้งานจริง (Utility-first) และต้องการความเร็วในการตอบสนอง (Real-time)
* **Simplicity:** ตัดสิ่งรบกวน (Distractions) ออกทั้งหมด เหลือไว้เฉพาะสิ่งที่จำเป็นต่องาน
* **Clarity:** สถานะของงาน (รอดำเนินการ/กำลังทำ/เสร็จสิ้น) และความเป็นเจ้าของงาน ต้องชัดเจนในเสี้ยววินาที
* **Invisible Interface:** ทำให้อินเทอร์เฟซดูธรรมดาที่สุด เพื่อให้ผู้ใช้โฟกัสไปที่ "ตัวงาน" ที่ต้องทำจริงๆ

---

## 2. การตัดสินใจด้านโครงสร้างและการจัดวาง (Layout Decisions)

### 2.1 Desktop-First Layout (1440px)
* **Rationale:** เลือกใช้ Layout มาตรฐาน Desktop เพราะ Use Case ของการทำงานร่วมกัน (Collaboration) มักเกิดขึ้นในสภาพแวดล้อมการทำงานบนหน้าจอคอมพิวเตอร์
* **Centered Container:** จำกัดความกว้างของเนื้อหาหลัก (Main Content) ไว้ที่กึ่งกลางหน้าจอ (Max-width: 800px) เพื่อให้อ่านง่าย (Readability) สายตาไม่ต้องกวาดมองกว้างเกินไปบนจอ Wide-screen

### 2.2 Single-Page Application (SPA)
* **Rationale:** แยกหน้า Login (Identify) ออกจาก Dashboard อย่างชัดเจน แต่เมื่อเข้าสู่ Dashboard แล้ว ทุกฟังก์ชัน (Add, Edit, Delete, Sync) จะจบในหน้าเดียว เพื่อลด Interaction Cost ไม่ต้องกดเปลี่ยนหน้าไปมา

### 2.3 การจัดวางตามหลักจิตวิทยา (Layout Mechanics)
* **F-Pattern Scanning:** จัดวางองค์ประกอบหน้า Dashboard เรียงจากบนลงล่าง (Header -> Input -> List) สอดคล้องกับธรรมชาติการกวาดสายตาของมนุษย์
* **Z-Layout (หน้า Login):** หน้า Login วางองค์ประกอบเรียบง่าย เพื่อนำสายตาผู้ใช้เป็นรูปตัว Z พุ่งเป้าไปที่ปุ่ม "Join Room" ทันที

## 3. การตัดสินใจด้านสีและส่วนติดต่อผู้ใช้ (Color & UI Decisions)

### 3.1 ทฤษฎีสี (Color Theory) และความหมาย
ใช้สีเพื่อสื่อความหมายของสถานะและปุ่มกด (Semantic Colors):
* **Blue (#2563eb):** สื่อถึงความสงบและความน่าเชื่อถือ ใช้กับ **ปุ่ม Add Task** และ **Header** เพื่อให้เป็นจุดโฟกัสหลัก
* **Yellow (#eab308):** สื่อถึงการแก้ไขหรือการระมัดระวัง ใช้กับ **ปุ่ม Edit** เพื่อให้ผู้ใช้สังเกตเห็นได้ง่ายแต่ไม่ดูรุนแรงเท่าสีแดง
* **Red (#dc2626):** สื่อถึงอันตรายและการลบทำลาย ใช้กับ **ปุ่ม Delete** เพื่อเตือนสติผู้ใช้ก่อนกด
* **Green (#16a34a):** สื่อถึงความสำเร็จ ใช้กับสถานะ **"เสร็จสิ้น" (Done)**

### 3.2 Visual Hierarchy (ลำดับความสำคัญทางสายตา)
* **Input Field:** ใช้ขนาดใหญ่และวางไว้บนสุด เพื่อเชิญชวนให้ผู้ใช้ "เริ่มสร้างงาน" เป็นอันดับแรก
* **Dropdown Status:** ใช้สีพื้นหลังที่แตกต่างกัน (เทา/เหลืองอ่อน/เขียวอ่อน) ในแต่ละสถานะ เพื่อให้ผู้ใช้แยกแยะสถานะงานได้ทันทีโดยไม่ต้องอ่านตัวหนังสือ

### 3.3 Typography (ตัวอักษร)
* เลือกใช้ฟอนต์ **Sans-serif** (เช่น Inter, Segoe UI) เพื่อความทันสมัยและอ่านง่ายบนหน้าจอดิจิทัล
* ใช้ **Strikethrough** (ขีดฆ่า) กับงานที่เสร็จแล้ว เพื่อลดความเด่นของงานนั้นลง ให้สายตาไปโฟกัสกับงานที่ยังไม่เสร็จแทน

## 4. การตัดสินใจด้านประสบการณ์ผู้ใช้และกระแสการใช้งาน (UX & Flow)

### 4.1 Real-time Feedback Loop
* **Design:** ไม่มีปุ่ม "Refresh" หรือ "Sync"
* **Rationale:** เพื่อสร้างความเชื่อมั่น (Trust) ว่าระบบทำงานแบบ Real-time อัตโนมัติ ลดภาระที่ผู้ใช้ต้องคอยกดโหลดข้อมูลเอง การเปลี่ยนแปลงจะเกิดขึ้นทันทีที่ผู้ใช้อีกคนกระทำ

### 4.2 การแก้ไขและลบ (Edit & Delete Interactions)
* **Design:** แยกปุ่ม Edit และ Delete ออกมาเป็นปุ่มกดชัดเจน (Button Style) แทนที่จะซ่อนไว้ในเมนู
* **Rationale:** เพื่อความรวดเร็วในการจัดการงาน (Efficiency) ผู้ใช้สามารถเข้าถึงฟังก์ชันสำคัญได้ในคลิกเดียว

### 4.3 บริบทความเป็นเจ้าของ (Ownership Context)
* **Design:** แสดงชื่อผู้สร้างและเวลาที่สร้าง (Timestamp) ใต้ชื่องาน
* **Rationale:** ในระบบ Multi-user การระบุตัวตนผู้สร้างงานช่วยสร้างความเกรงใจและลดความขัดแย้ง (Conflict) ในการลบหรือแก้ไขงานของผู้อื่น

---

## 5. การจัดการข้อจำกัด (Handling Constraints)
* **Conflict Resolution:** กรณีมีการแก้ไขงานเดียวกันพร้อมกัน UI จะยึดตามข้อมูลล่าสุด (Last Write Wins) โดยไม่มี Pop-up แจ้งเตือนที่ซับซ้อน เพื่อรักษาความลื่นไหลของระบบ (Simplicity over Complexity) ตามขอบเขตของ MVP
